# AGENTS.md Bootstrap Generation

Date: 2026-04-27

## Decision

Generated `AGENTS.md` files are compatibility/bootstrap exports only. Shared
agent policy lives in Den documents and first-class Den guidance entries.

The generator remains in `den-mcp` because it knows how to find Den-registered
project roots and emit a consistent bootstrap file for tools that still read
`AGENTS.md`.

## Source Of Truth

Global shared policy is stored in `_global` Den documents and included through
global guidance entries. Resolve the live packet with:

```text
get_agent_guidance(project_id: "<project-id>")
```

or:

```bash
den guidance --project <project-id>
```

The generated `AGENTS.md` tells agents to resolve that Den guidance and to stop
if Den is unavailable. It must not duplicate global workflow policy.

## Source Layout

Each opted-in project keeps one local bootstrap fragment at its repo root:

```text
agents-project.md
```

The local fragment is plain Markdown containing only project-specific bootstrap
context:

- project overview or mission
- project-specific architecture or invariants
- build, run, or test commands useful before deeper guidance is loaded
- repo-specific conventions that do not belong in global Den policy

The old `den-mcp/agents/shared/*.md` fragment set has been retired. Shared
policy edits should update Den documents and guidance entries instead.

## Generation Model

`scripts/generate_agents.py` composes:

1. An autogenerated warning header.
2. A Den guidance bootstrap section naming the project id and guidance lookup.
3. The project-local `agents-project.md` fragment.

The output is written to `AGENTS.md` in the target project root.

The output is deterministic:

- fixed bootstrap text
- normalized `\n` newlines
- trimmed fragment boundaries
- stable autogenerated header text

For opted-in projects, the generated `AGENTS.md` is intended to be committed so
a fresh clone has enough guidance to connect to Den or stop cleanly when Den is
unavailable.

## Target Selection

V1 targets all Den-registered projects with a `root_path`, but generation is
opt-in by local fragment presence.

Behavior:

- `--root /path/to/repo` generates one project directly. The project id is the
  root directory name.
- `--project-id den-mcp` resolves the root from the Den SQLite database and uses
  the Den project id.
- `--all` scans all registered projects in Den and generates `AGENTS.md` only
  for projects that contain `agents-project.md`.

Projects without `agents-project.md` are skipped rather than treated as errors
during `--all`. This keeps rollout incremental.

## Commands

Generate for `den-mcp` from the current checkout:

```bash
python3 scripts/generate_agents.py --root /home/patch/dev/den-mcp
```

Generate by Den project id:

```bash
python3 scripts/generate_agents.py --project-id den-mcp
```

Generate every opted-in registered project:

```bash
python3 scripts/generate_agents.py --all
```

Check whether generated files are up to date without rewriting:

```bash
python3 scripts/generate_agents.py --all --check
```

## Editing Workflow

For shared policy edits:

1. Update the relevant Den document, usually under `_global`.
2. Add or update the guidance entry when inclusion, importance, audience, or
   order changes.
3. Verify with `get_agent_guidance(project_id: "<project-id>")`.
4. Regenerate `AGENTS.md` only when the bootstrap/export text or local project
   fragment changed.

For project-local bootstrap edits:

1. Edit `agents-project.md` in that project.
2. Run the generator for that project.
3. Commit both the local fragment and generated `AGENTS.md`.

## Scope

The generator manages `AGENTS.md` only.

It intentionally does not manage sibling prompt files or other repo-local agent
assets.
