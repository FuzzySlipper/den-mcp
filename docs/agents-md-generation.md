# AGENTS.md Generation

Date: 2026-04-23

Status update, 2026-04-26: generated guidance now treats dispatches as legacy/debug artifacts rather than normal startup/drain work; see [ADR: Retire dispatches from the canonical conductor workflow](dispatch-retirement-adr.md).

## Decision

Keep the AGENTS generator and the shared source-of-truth fragments in `den-mcp` directly.

Rationale:
- The workflow guidance already lives with Den concepts such as tasks, review packets, agent handoffs, agent-stream attention, and legacy dispatch compatibility notes.
- The first consumers are existing Den-managed repos on the same machine, so a shared package would add distribution overhead before the format is stable.
- A simple script in `den-mcp` is enough to validate the model before considering extraction.

## Source Layout

Shared fragments live in this repo:

```text
agents/
  manifest.json
  shared/
    task-management.md
    review-loop.md
    agent-expectations.md
    git-workflow.md
    code-quality.md
```

Each opted-in project keeps one local fragment at its own repo root:

```text
agents-project.md
```

The local fragment is plain Markdown containing only project-specific guidance. It should not contain an autogenerated warning header.

Minimal local fragment shape:
- Project overview or mission.
- Project-specific architecture or invariants.
- Project-specific build, run, or test instructions.
- Repo-specific conventions that do not belong in shared workflow guidance.

## Generation Model

`scripts/generate_agents.py` composes:
1. An autogenerated warning header.
2. The project-local `agents-project.md`.
3. The ordered shared fragments from `agents/manifest.json`.

The output is written to `AGENTS.md` in the target project root.

The output is deterministic:
- fixed fragment order from `agents/manifest.json`
- normalized `\n` newlines
- trimmed fragment boundaries
- stable autogenerated header text

For opted-in projects, the generated `AGENTS.md` is intended to be committed so a fresh clone already contains the guidance that agent tools expect. The editable source of truth is still the fragment set, not the generated file.

## Target Selection

V1 targets all Den-registered projects with a `root_path`, but generation is opt-in by local fragment presence.

Behavior:
- `--root /path/to/repo` generates one project directly.
- `--project-id den-mcp` resolves the root from the Den SQLite database and generates that project.
- `--all` scans all registered projects in Den and generates `AGENTS.md` only for projects that contain `agents-project.md`.

Projects without `agents-project.md` are skipped rather than treated as errors during `--all`. This keeps rollout incremental instead of forcing every registered repo to migrate at once.

## Commands

Generate for `den-mcp` from the current checkout:

```bash
python3 scripts/generate_agents.py --root /home/patch/dev/den-mcp
```

Generate by Den project id:

```bash
python3 scripts/generate_agents.py --project-id den-mcp
```

Generate every opted-in registered project:

```bash
python3 scripts/generate_agents.py --all
```

Check whether generated files are up to date without rewriting:

```bash
python3 scripts/generate_agents.py --all --check
```

## Migration Status

Validated in v1 by migrating `den-mcp` itself:
- project-local content moved to `agents-project.md`
- shared workflow guidance moved to `agents/shared/*.md`
- root `AGENTS.md` is now generated from those sources

Other registered repos can migrate incrementally by adding `agents-project.md` and then running the same generator.

## Scope

V1 covers `AGENTS.md` only.

The generator intentionally does not manage sibling prompt files or other repo-local agent assets yet. Those can be added later if the fragment model proves stable.
